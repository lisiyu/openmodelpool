# ModelMux v3.0 路线图

## 核心愿景：P2P Provider 共享联邦

多个独立部署的 modelmux 实例之间自动发现、共享 Provider，组成去中心化的 AI 模型集群。用户选择"愿意共享"后，自己的 Provider 以"中继模式"贡献给联邦网络，API Key 永远不出本机。

---

## Phase 1：联邦基础架构

### 1.1 节点身份与联邦协议
- 每个 modelmux 实例生成唯一 NodeID（Ed25519 密钥对）
- 节点间通信基于 mTLS（双向 TLS 认证）
- 定义联邦消息格式（JSON over HTTPS）：
  - `announce`：宣告自己共享的模型列表
  - `probe`：探测远程节点可用性
  - `relay`：转发请求到远程节点

### 1.2 Provider 中继模式
- 共享 Provider 时，不是暴露 API Key，而是暴露一个「中继端点」
- 请求链路：用户 → 本地 modelmux → 远程 modelmux → 实际 AI API
- 远程节点用自己的 Key 调用实际 API，结果原路返回
- 中继节点可设置：
  - 是否允许匿名使用
  - 速率限制（req/min）
  - 白名单（仅信任节点可用）
  - 是否需要积分

### 1.3 节点发现（Registry）
- 轻量中心 Registry 服务（可选自部署）
  - 节点注册：上报自己的公网地址 + 共享模型列表
  - 节点查询：按模型名搜索哪些节点提供该模型
  - 心跳维护：定期上报存活状态
- 备选方案：mDNS（局域网）/ DHT（去中心化，类 BitTorrent）

## Phase 2：信任与激励

### 2.1 信任网络
- Web of Trust 模型：节点间互相签名背书
- 信任传递：A 信任 B，B 信任 C → A 可间接信任 C
- 信誉评分：基于成功率、延迟、在线时长自动计算

### 2.2 积分/配额系统
- 共享 Provider 获得积分（按实际调用量）
- 使用他人的 Provider 消耗积分
- 积分可跨节点转移
- 防止滥用：单节点每日最大消耗上限

### 2.3 计费与成本分摊
- 中继节点可设定每次调用的"成本"（积分或实际金额）
- 成本透明：路由时展示每个候选节点的成本
- 可选：集成 Lightning Network 微支付

---

## Phase 3：智能联邦路由

### 3.1 联邦路由引擎
- 扩展现有 4 维路由到 5 维：+ 节点信任度
- 跨节点 EWMA 延迟评估
- 自动选择最优中继路径（最小化跳数 + 延迟）

### 3.2 请求调度优化
- 请求级缓存：相同 prompt 命中缓存直接返回
-  speculative routing：同时向多个节点发起请求，取最快响应
- 连接池复用：节点间保持长连接减少握手开销

### 3.3 数据一致性
- 模型名称标准化：不同平台的同名模型自动映射
- 能力标签：模型支持的能力（function calling、vision、code）统一标注

---

## Phase 4：隐私与安全

### 4.1 请求隐私
- 中继节点无法窥探请求内容（可选 E2E 加密模式）
- 盲中继模式：中继只知道模型名，不知道 prompt 内容
- 审计日志：节点可记录谁在什么时间调用了什么

### 4.2 网络安全
- DDoS 防护：节点级别的速率限制 + 流量整形
- 恶意节点检测：自动摘除异常节点（高失败率、篡改响应）
- 证书吊销：CRL/OCSP 自动检查

---

## Phase 5：生态与体验

### 5.1 联邦管理面板
- 可视化联邦拓扑图（节点关系、流量热力图）
- 一键加入/退出联邦
- 共享策略配置（哪些 Provider 共享、共享给谁）

### 5.2 插件系统
- 自定义 Provider 适配器（通过 WASM 插件扩展）
- 自定义路由策略插件
- Webhook 事件通知

### 5.3 移动端 & CLI
- 手机 App 监控面板
- CLI 工具管理联邦节点
- Terraform provider 自动化部署

## 技术选型建议

| 组件 | 方案 | 理由 |
|------|------|------|
| 节点通信 | gRPC over TLS | 高性能、强类型、流式支持 |
| 节点发现 | 自建 Registry + mDNS 备选 | 简单可靠，局域网自动发现 |
| 身份认证 | Ed25519 + mTLS | 无中心 CA，去中心化信任 |
| 数据同步 | CRDT | 无冲突最终一致性 |
| 服务发现 | libp2p (可选) | 如果需要完全去中心化 |
| 消息队列 | NATS (可选) | 轻量级节点间事件广播 |

## 与 v2.0 的关系

v2.0 的多用户架构为 v3.0 奠定了关键基础：
- **Consumer/Provider 隔离模型** → 扩展到节点间隔离
- **API Key 认证体系** → 扩展到节点身份认证
- **路由引擎（4维）** → 扩展到联邦路由（5维+跨节点）
- **健康检测** → 扩展到联邦节点存活探测
- **Token 预算** → 扩展到联邦积分系统
